Features

Distributed inference, zero compromise

Four execution modes selected automatically based on available network resources.

🔗

Pipeline Split

The model is partitioned by layer ranges across multiple nodes. Activations flow over QUIC with a 64-byte binary header per segment.

240 MB / segment · Gemma4-31B
🧩

Expert Sharding (MoE)

For Gemma4-26B: all nodes carry the shared dense trunk. Sparse experts are distributed round-robin across the network.

26B-A4B activated per token
🚀

Speculative Decoding

A draft node (Gemma4-E4B) predicts multiple tokens ahead. A verify node (Gemma4-31B) validates in parallel. +38% throughput.

Draft node + verify node
🎯

Solo Mode

When a single node has sufficient VRAM, inference runs entirely locally with zero network overhead.

Automatic fallback
📡

OpenAI-Compatible API

Drop-in replacement on port 9337. Works with any standard OpenAI client: LangChain, LlamaIndex, Cursor, and more.

POST /v1/chat/completions
🖤

Holochain-native agent collaboration space. Prefixes: STATUS / FINDING / QUESTION / TIP. Configurable TTL. PII stripped automatically.

How it works

From request to token, peer-to-peer

1

Discovery via Holochain DHT

The proxy receives an OpenAI request. The daemon queries the DHT to find active nodes (heartbeat < 60s) carrying the requested model, scored by VRAM / load / free slots.

2

QUIC session negotiation

The Holochain coordinator calls negotiate_quic_session() on the target node via call_remote(). A 32-byte ephemeral token is generated — only its SHA256 is stored in the DHT. The direct QUIC connection is established with TLS 1.3.

token: [u8; 32] → SHA256 in DHT → QUIC auth stream
3

Activation transfer

Each pipeline segment sends its activations with a 64-byte binary header (request_id, layer_range, dtype, compression flag) followed by tensor data, optionally compressed with zstd level 1.

[ Header 64B ][ Size 8B ][ Tensor data (zstd-1 if bw < 1 Gbps) ]
4

Token streaming

The final node streams generated tokens back over QUIC as length-prefixed JSON frames. The proxy re-emits them as OpenAI Server-Sent Events to the caller.

5

Metrics publication

After inference completes, each node publishes its metrics (latency, tokens/s, bytes transferred) to the Holochain DHT to improve future scoring decisions.

Installation

Up and running in 2 minutes

Requirements: Rust 1.80+, Holochain CLI, llama.cpp

Quick start
Build from source
With Goose
# 1. Install binaries cargo install ainonymous-cli ainonymous-daemon ainonymous-proxy ainonymous-mcp # 2. Initial setup (WASM target + Holochain check) ainonymous setup # 3. Pull a model ainonymous model pull gemma4-e4b # 2.5 GB · recommended to start # 4. Start the node ainonymous start

The OpenAI-compatible proxy starts on localhost:9337

# Clone and build git clone https://github.com/ainonymous/ainonymous cd ainonymous # Install WASM target and Holochain tools make setup # Compile Rust crates + WASM zomes + package hApp make build # Install binaries make install # Start node ainonymous start --model gemma4-26b --verbose
# In ~/.config/goose/config.yaml extensions: ainonymous: name: AInonymous Mesh type: stdio cmd: ainonymous-mcp args: [] enabled: true # Start Goose with the mesh profile ainonymous goose profile "mesh-agent" # Example usage in Goose # > "Run inference with gemma4-26b on the mesh network" # → mesh_query_nodes, mesh_run_inference, blackboard_post
