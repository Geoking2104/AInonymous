Pourquoi une boucle fermée change tout

Dans un déploiement public, vos données traversent des serveurs tiers — anonymisées ou non. Dans la boucle fermée AInonymous, la question d'un collaborateur ne quitte jamais votre LAN ou VPN d'entreprise.

La Distributed Hash Table (DHT) Holochain tourne à l'intérieur de votre infrastructure. Les nœuds se découvrent mutuellement par votre réseau interne, et les échanges sont chiffrés de bout en bout par des clés ed25519 propres à chaque machine.

  • ✓Découverte de nœuds : via le réseau d'entreprise (pas de relais externe Nostr ou public)
  • ✓Transport : WebRTC sur LAN ou QUIC sur VPN — pas d'internet requis
  • ✓État partagé : source chain Holochain immutable, auditée, locale
  • ✓Identité : Membrane Proofs pour contrôler qui rejoint le réseau privé
  • ✓Air-gap possible : fonctionne sur réseau isolé sans accès internet

Infrastructure P2P

Holochain — DHT agent-centrique

Pas de blockchain, pas de consensus global. Chaque nœud maintient sa propre source chain immuable. La DHT valide l'état partagé sans serveur central.

Holochain 0.4 · Rust · WebRTC · ed25519
Inférence distribuée

Pipeline-splitting par couches

Un modèle 31B trop grand pour une seule machine est découpé par couches et distribué sur plusieurs nœuds. Le décodage spéculatif (draft + verify) accélère l'inférence.

llama.cpp · GGUF · layer-sharding · spec-decode
Sécurité & identité

Membrane Proofs + Warrants

Seules les machines détenant une Membrane Proof valide rejoignent le réseau privé. Les warrants cryptographiques attestent la réputation et les capacités de chaque nœud.

ed25519 · zomes de validation · warrants Holochain
Compatibilité

API OpenAI-compatible locale

Endpoint localhost:9337/v1 reproduit l'API d'OpenAI. Copilot, LangChain, n8n, Cursor, Goose — tous fonctionnent sans modification.

REST · SSE streaming · OpenAI v1 spec
Orchestration agents

Goose (Block, Apache 2.0) orchestre des agents IA complexes via MCP natif. Ses serveurs MCP se connectent aux zomes Holochain pour accéder au blackboard partagé.

Comment une requête traverse le réseau

Du prompt de votre collaborateur à la réponse — sans jamais toucher l'extérieur.

1

Prompt utilisateur

L'utilisateur envoie une requête via son outil (VS Code, chat interne, n8n…)

2

API proxy locale

Le proxy :9337 reçoit la requête OpenAI-compat. et l'envoie au conducteur Holochain

3

Routage DHT

Le mesh consulte le registre des nœuds pour trouver les machines disponibles selon le modèle et la charge

4

Inférence distribuée

Les couches du modèle sont exécutées en pipeline sur les nœuds sélectionnés (layer-sharding)

5

Streaming tokens

Les tokens générés sont streamés vers l'API en temps réel — latence identique à du local

6

Audit chain

La tâche est chainée cryptographiquement pour traçabilité. Aucune donnée sensible stockée.

Chaque couche du système est pensée pour satisfaire les exigences des DSI, DPO et RSSI.

🔑

Identités cryptographiques

Chaque nœud est identifié par une paire de clés ed25519. Aucune authentification centralisée, aucun mot de passe — la cryptographie est le seul mécanisme d'identité.

ed25519 · Holochain keys
🚪

Contrôle d'accès au réseau

Les Membrane Proofs définissent qui peut rejoindre le réseau privé. Un ordinateur sans proof valide est invisible pour le mesh — même sur le même LAN.

Membrane Proofs · RBAC
🔒

Chiffrement de bout en bout

Tous les échanges entre nœuds sont chiffrés. Les données en transit sont protégées même sur un réseau interne compromis.

TLS · WebRTC DTLS · E2E
📋

Audit trail immuable

Chaque tâche d'inférence est ajoutée à une source chain Holochain — une structure append-only cryptographiquement liée. Idéal pour les audits réglementaires.

Source chain · append-only
🌐

Isolation réseau totale

Le réseau peut tourner sans aucun accès internet. Modes supportés : LAN interne, VPN d'entreprise, réseau air-gappé. Aucune dépendance à des relais ou DNS externes.

Air-gap ready · VPN · LAN
🔍

Validation des pairs

Les zomes de validation Holochain permettent à chaque nœud de vérifier la conformité des données des autres avant de les accepter. Pas de nœud malveillant non détecté.

Zome validation · warrants

Modes de déploiement

Du pilote à l'échelle groupe.

Démarrez petit, scalez sans migration — l'architecture est identique à 5 ou 5 000 nœuds.

🌱

Pilote équipe

10–30 machines, réseau LAN. Idéal pour valider le cas d'usage sur une équipe tech ou R&D. Déploiement en moins d'une journée.

Gemma4-E4B LAN 1 jour
🏢

Déploiement site

50–500 machines, VPN d'entreprise. Modèles 26B–31B distribués, multi-services, API centralisée par site. Configuration recommandée pour la majorité des entreprises.

Gemma4-31B VPN · LAN 1–2 semaines

Groupe multi-sites

500+ machines, multi-sites, réseau maillé entre filiales. Chaque site dispose d'un sous-réseau autonome avec fédération possible. Support Llama 3.3 70B.
